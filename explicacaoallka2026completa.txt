1. O QUE É A ALLKA (visão de negócio)
================================================================================
A Allka é uma plataforma de marketplace de serviços que conecta três tipos de
elos numa cadeia de prestação de serviço:

  EMPRESA (cliente final)  →  AGÊNCIA (opera/revende)  →  NÔMADE (executa)

- EMPRESA ("company"/"empresa"): o cliente que contrata serviços/produtos
  (ex.: tráfego pago, design, desenvolvimento) através de projetos.
- AGÊNCIA ("agency"/"agencia"): reveste/opera a entrega para o cliente, monta
  o time, acompanha financeiro e projetos. Agências têm um nível de parceria
  (Bronze/Silver/Gold/Platinum/Diamond) dentro do "Programa Partner".
- NÔMADE ("nomades"/"nomad"): o especialista freelancer que efetivamente
  executa as tarefas (ex.: um designer, um redator, um dev). Também tem
  níveis de gamificação (Bronze→Diamond/Leader) baseados em performance.
- LÍDER ("leader"/"lider"): um papel interno de qualidade — cada líder cuida
  de uma "área" (categoria de especialidade) e aprova/qualifica o trabalho
  dos nômades antes de liberar para o cliente.
- PARCEIRO ("partner"/"parceiro"): um afiliado/indicador que traz agências ou
  empresas para a plataforma e ganha comissão recorrente sobre o MRR gerado
  (ex.: 5% do MRR das agências que ele "lidera").
- ADMIN: a equipe interna da Allka, dona da operação — cadastra produtos,
  define preços/comissões, audita tudo, cuida do financeiro global.

O núcleo operacional é: uma EMPRESA contrata um PROJETO composto por
PRODUTOS do catálogo; cada produto gera TAREFAS (a partir de "modelos de
tarefa"); as tarefas são executadas por NÔMADES qualificados automaticamente
por especialidade; um LÍDER de área revisa/aprova antes da entrega final;
tudo isso gera FATURAS, COMISSÕES e movimentações de CARTEIRA (wallet) no
financeiro.

2. ARQUITETURA TÉCNICA
================================================================================

MONOREPO com duas pastas principais em apps/:
  apps/backend   → API (Node.js + TypeScript + Express 5 + Prisma ORM)
  apps/frontend  → SPA (Vite + React + react-router-dom)

⚠️ DETALHE IMPORTANTE (fácil de confundir): apesar da estrutura de pastas
`apps/frontend/app/<secao>/<pagina>/page.tsx` parecer Next.js App Router,
o frontend NÃO usa Next.js. É uma SPA Vite com roteamento manual feito em
`apps/frontend/App.tsx`, que importa cada `page.tsx` via `React.lazy()` e
registra rotas explícitas com `<Route path="...">` do react-router-dom. Isso
significa que a URL de uma página e o caminho físico do arquivo no disco
NEM SEMPRE coincidem (existem pastas cujo conteúdo está "morto"/não roteado,
e páginas cuja URL usa uma pasta enquanto o código real mora em outra pasta
com nome parecido em português). Isso está detalhado na seção 15.

BANCO DE DADOS: MySQL, acessado via Prisma ORM. Schema completo em
  apps/backend/prisma/schema.prisma

AUTENTICAÇÃO: JWT stateless (sem sessão/cookie), token válido por 7 dias.
Detalhes completos na seção 12.

INTEGRAÇÕES EXTERNAS REAIS: praticamente nenhuma ainda. O gateway de
pagamento é 100% sandbox/fake (`FAKE_SANDBOX`), sem Stripe/Pagar.me/Asaas
real. Não há envio de e-mail, WhatsApp ou SMS real. Não há Bitrix real (só
um campo `bitrix_sync` reservado). Detalhes na seção 11.

3. PERFIS DE USUÁRIO E COMO CADA UM SE LOGA
================================================================================

Existem 6 portais logados + 1 página pública de compartilhamento:

  /admin/login     → Admin (equipe interna Allka)
  /company/login   → Empresa/Cliente
  /agency/login    → Agência
  /leader/login    → Líder de área
  /nomad/login     → Nômade (freelancer)
  /partner/login   → Parceiro/Afiliado

Cada login usa o mesmo template visual (`LoginPageTemplate`, split-screen com
texto de marketing multilíngue pt/en/es/zh à esquerda e formulário à
direita), mudando só a tag de marca ("Allka Company", "Allka Agency", "Allka
Líder", "Nômades Allka", "Allka Partner") e o `accessType` enviado no login
(ADMIN/COMPANY/AGENCY/LEADER/NOMAD/PARTNER), que o backend valida contra o
`role`/`account_type` real do usuário.

Depois de logado, TODOS os perfis (menos o link público de compartilhamento)
caem dentro do MESMO layout global (`AppLayout` em App.tsx): mesmo
Sidebar, mesmo Header, mesmo Footer, mesmo ChatWidget flutuante — só o
CONTEÚDO do menu lateral e das páginas muda de acordo com o perfil. Não há
"shells" visuais separados por papel (existe um `parceiro/layout.tsx` com
sidebar própria, mas está morto/não usado — ver seção 15).

Guarda de acesso (`RequireAuth` em App.tsx):
- Sem token em localStorage → redireciona para o /login do prefixo da URL
  atual (ex.: tentar acessar /agency/* sem estar logado manda para
  /agency/login).
- Com token, mas o `role`/`account_type` do usuário não bate com o prefixo
  da URL que ele está tentando acessar → é jogado de volta para o dashboard
  do PRÓPRIO perfil (ex.: um nômade tentando abrir /company/dashboard é
  redirecionado para /nomad/dashboard).
- Admin (`role==="admin"` ou `account_type==="admin"`) pode acessar
  qualquer área.
- Um evento global `allka:unauthorized` (disparado quando qualquer chamada
  de API retorna 401) redireciona automaticamente para o login certo.
- Existe também um `TermAcceptanceGate` — uma tela bloqueante que aparece
  para QUALQUER perfil se houver termos de uso pendentes de aceite.

MODO DEV/PREVIEW: há um mecanismo (`AccountTypeContext` +
`dev-role-switcher.tsx`) que permite trocar de perfil simulado via
localStorage (`dev_active_profile`) sem precisar logar de verdade — útil
para desenvolvimento, mas não é o fluxo de produção real.

4. PORTAL DA EMPRESA (company/empresa)
================================================================================
⚠️ Nota de duplicação: existe uma pasta `app/empresa/*` e uma pasta
`app/company/*` quase idênticas. A pasta REAL/ativa (a que realmente é
servida nas URLs) é `company/*`. A pasta `empresa/*` está 100% órfã — nenhum
arquivo dela é importado em lugar nenhum do app. Ignore `empresa/*`.

Contexto de dados: `useEmpresa()` (contexts/empresa-context.tsx) — carrega
perfil, projetos (com produtos/etapas), tarefas e faturas da empresa logada.

4.1) company/dashboard — "Dashboard da Empresa"
  Dashboard 100% customizável por widgets (mesmo framework usado por
  Agency/Leader/Partner — dashboard-presets-by-role.ts).
  Widgets disponíveis: métricas gerais, projetos ativos, tarefas, visão de
  status, atividade recente, alertas, ações rápidas. (Empresa NÃO vê
  widgets financeiros/administrativos de toda a plataforma como MRR/LTV/
  CMV/CHURN, distribuição de usuários ou matriz de permissões — esses são
  exclusivos do admin.)
  Views pré-configuradas (como "abas" de dashboard): Visão Geral da Company,
  Projetos Contratados, Aprovações, Financeiro e Pagamentos, Entregas e
  Histórico.
  Ações: personalizar widgets (adicionar/editar/ocultar/remover), ver
  detalhes de um widget, exportar widget (PNG), compartilhar widget ou o
  dashboard inteiro (gera link público — ver seção sobre /dashboard/share),
  salvar/criar/editar/excluir dashboards salvos, "seguir período global"
  (segue o filtro de data escolhido), criar métricas manuais por mês/ano.

4.2) company/projetos — "Gestão de Projetos"
  Board central de projetos contratados. Visualização em Kanban com colunas
  editáveis (criar/renomear/excluir coluna). "Contratar Projetos" abre o
  catálogo para montar um novo projeto. Rascunhos de projeto podem ser
  continuados ("Continuar rascunho") ou pagos ("Ir para pagamento").
  Cancelar um projeto dispara um fluxo de retenção (telas de "Esperamos que
  continue conosco!" / "Última chance para reconsiderar" antes de confirmar
  o cancelamento). Tem exportação (CSV/Excel/PDF) e cartões de estatística
  (Total de Projetos, Em Andamento, Concluídos).

4.3) company/tarefas
  Lista de tarefas de todos os projetos da empresa, com filtro por status
  (Disponível / Em execução / Em revisão / Concluída / Cancelada) e busca
  por nome de tarefa/projeto/nômade.

4.4) company/faturas
  Histórico de faturas (número, descrição, valor, emissão, vencimento,
  status: Pendente/Paga/Vencida/Cancelada), com filtro por status e cartões
  de resumo (Pendentes, Pagas, Total Geral).

4.5) company/produtos — "Catálogo de Produtos"
  Vitrine dos produtos/serviços contratáveis (componente compartilhado
  `ProductCatalogView`, o mesmo usado em Agência/Líder). Carrinho
  (`useProjectBasket`) com add/remove/aumentar/diminuir quantidade. Botão
  "Contratar Serviços" abre o painel de criação de projeto para fechar a
  contratação a partir do carrinho.

4.6) company/relatorios
  Diferente das outras 4 (agency/leader/partner/nomades), a Empresa tem sua
  PRÓPRIA página de relatórios (não usa o componente genérico compartilhado
  `ReportListPage`). Categorias: Financeiro (Minhas Faturas, Investimentos
  por Projeto, Fluxo de Pagamentos), Projetos e Entregas (Status, Progresso,
  Timeline/Prazos), Tarefas e Atividades (por status, performance,
  atividade da equipe). Tem seletor de período (calendário), busca, e
  ações "visualizar"/"baixar" (PDF/XLSX) por relatório.

4.7) company/login — tela de login (ver seção 3).

── /dashboard/share/:token (fora de qualquer portal — página pública) ──────
Não é um "hub" nem um seletor de perfil: é o VISUALIZADOR de links de
compartilhamento gerados pelo botão "Compartilhar Dashboard"/"Compartilhar
widget" que existe dentro de cada dashboard (Empresa/Agência/Líder/
Parceiro/Nômade). É uma rota PÚBLICA (sem autenticação, fora do
AppLayout/Sidebar/Header). O token codifica: alvo (widget específico ou
dashboard inteiro), perfil dono, nível de permissão (somente visualizar,
permitir trocar filtros, ou comentar), PIN opcional e validade opcional.
Se protegido por PIN, mostra tela de "Acessar relatório"; se expirado ou
inválido, mostra tela de erro dedicada; se a permissão for "comment",
mostra uma caixa de comentário. Só é alcançado por link gerado, nunca por
navegação dentro do app.

10. MODELO DE DADOS (banco de dados / Prisma)
================================================================================
Fonte: apps/backend/prisma/schema.prisma — banco MySQL, IDs `cuid()`.

USUÁRIOS E AUTENTICAÇÃO
- User: identidade/login central. role (company_user | company_admin |
  company_financial | agency_user | agency_admin | nomad | admin |
  partner), account_type (empresas | agencias | nomades | admin |
  parceiro), company_id opcional. Relaciona 1:1 opcional com Nomade,
  Agency, PartnerProfile; também com TermAcceptance, ChatParticipant,
  ChatMessage, CourseEnrollment, LiderArea, ProjectTask (como
  lider_responsavel), Coupon (como linked_user).
- LiderArea: vincula um User (líder) a um nome de área + categorias/
  produtos permitidos (JSON), usado para auto-atribuir tarefas de revisão.
- AdminProfile / AdminPermission: perfis de permissão nomeados (com flag
  is_master) e permissões por módulo/ação (view/edit/create/delete/
  not_applicable).

EMPRESAS / AGÊNCIAS / CONTAS
- Company: empresas/clientes. sequence_number (código de exibição
  emp_00001), cnpj único, type (empresa/agencia/nomade/parceiro — a
  classificação comercial real), status (ativo/inativo/prospecto/
  inadimplente), blocos de contato comercial/financeiro,
  referred_by_partner_id → PartnerProfile. Tem muitos Project, Invoice,
  User, CompanyPaymentMethod; 1:1 SquadConfig; muitos CouponUsage.
- CompanyPaymentMethod: cartões mascarados por empresa (bandeira, últimos
  4 dígitos, se é cartão do cliente ou da agência).
- Agency: 1:1 com User. partner_level (bronze/silver/gold/platinum/
  diamond), wallet_balance. Tem 1:1 MatchQueueEntry, muitos
  AgencyLeadership, muitos AgencyReport.

ESPECIALIDADES / NÔMADES
- Nomade: perfil de freelancer, User 1:1 opcional. level (bronze→leader),
  status (ativo/inativo/aguardando_aprovacao/reprovado/pausado), score,
  contadores de tarefa, is_leader/leader_id, métricas de performance
  (nota média, % no prazo, taxa de rejeição). Relaciona Qualification[],
  WalletTransaction[], BankAccount (1:1), TaskExecution[],
  WithdrawalRequest[], NomadeHabilidade[].
- NomadeHabilidade: vínculo estruturado de habilidade (área/categoria de
  produto/produto/modelo de tarefa) com nota média e disponibilidade —
  usado pelo algoritmo de auto-atribuição.
- NomadeLevel: tabela de níveis (slug, score mínimo/máximo, benefícios/
  requisitos em JSON).
- Qualification: certificação legada por categoria/tipo de tarefa
  (habilitado/pausado/teste_pendente/reprovado).
- WalletTransaction: lançamento legado de carteira do nômade — em grande
  parte substituído pelos modelos Wallet/WalletLedger mais novos.
- BankAccount: dados bancários/PIX do nômade para repasse.
- Specialty: catálogo de categorias de especialista com hourly_rate.

PROGRAMA PARTNER E NÍVEIS
- PartnerProfile: 1:1 com User; a conta "parceiro" de indicação/afiliado.
  balance, total_earned, total_withdrawn, referral_code único, status
  (active/suspended/pending), dados de pix. Relaciona
  PartnerCommission[], AgencyLeadership[] (agências lideradas),
  AgencyReport[], Company[] (empresas indicadas).
- PartnerLevel: tabela de níveis de parceiro (limiares de MRR, taxa de
  comissão, desconto extra, requisitos de agências lideradas, bônus de
  subida de nível).
- AgencyLeadership: vínculo entre um PartnerProfile e uma Agency que ele
  "lidera" (mentoria/gestão), com status/notas/período.
- AgencyReport: relatório mensal que um parceiro escreve sobre uma agência
  liderada (avaliação, destaques/melhorias em JSON, MRR, contagem de
  projetos/tarefas, rascunho/publicado).
- MatchQueueEntry: 1:1 com Agency; fila/tier (basic/premium/elite) +
  pontuação/posição para casamento de leads.

CAMPANHAS, INDICAÇÕES E CUPONS
- Campaign: campanha de marketing (tipo cupom/link/indicação,
  tipo/valor de comissão, código de cupom único opcional). Tem muitos
  PartnerCommission.
- Coupon: cupom de desconto/crédito/indicação (tipo, tipo/valor de
  desconto, bônus de crédito, limites de uso, tipos de conta/empresas/
  usuários permitidos em JSON, usuário vinculado opcional para comissão de
  indicação). Tem muitos CouponUsage.
- CouponUsage: registro de resgate ligando um Coupon a uma Company.
- PartnerCommission: comissão ganha por um parceiro a partir de uma
  campanha (valor, status pendente/aprovada/paga).

PROJETOS
- Project: registro central de engajamento. client_id → Company, status
  (draft→...→paid), lifecycle (avulso/mensal), valor/orçamento/gasto/
  progresso, flag bitrix_sync (só o campo — sem integração real por
  trás), billing_day. Relaciona TaskExecution[], Invoice[],
  ProjectProduct[], ProjectTask[], Payment[], ProjectCredential[].

PRODUTOS / CATÁLOGO
- Product: serviço/pacote vendável (categoria, preço base, complexidade,
  visibilidade por tipo de conta em JSON, metadata JSON). Tem
  ProductVariation[], ProductAddon[], ProductCatalogTask[] (vínculos de
  tarefa), ProjectProduct[], ProjectTask[].
- ProductVariation: nível de preço/escopo de um produto (preço, prazo em
  dias, features em JSON).
- ProductAddon: item adicional opcional precificado de um produto.
- CatalogTask: definição mestre de "modelo de tarefa" (código CT-xxx,
  categoria, tipo de tarefa, etapas/checklist/perguntas de briefing em
  JSON, regras de execução) — o template a partir do qual tarefas
  operacionais são geradas.
- ProductCatalogTask: junção N:N Product↔CatalogTask com overrides por
  vínculo (ordem, obrigatória, fase).

PROJETOS × PRODUTOS × TAREFAS (camada operacional de execução)
- ProjectProduct: um Product contratado dentro de um Project; carrega um
  SNAPSHOT comercial (nome/código/categoria/preço no momento da venda,
  comissão, tipo de pagador AGENCIA/CLIENTE), status (PENDENTE/
  EM_EXECUCAO/CONCLUIDO/CANCELADO). Tem ProjectTask[], ProjectCredential[].
- ProjectTask: a tarefa operacional real gerada a partir de um
  CatalogTask para um projeto/produto. task_code globalmente único
  (T000001…), pipeline de status rico (PARA_LANCAMENTO→...→CONCLUIDA/
  CANCELADA/AGUARDANDO_NOMADE), campos de atribuição
  (nomade_responsavel_id, lider_responsavel_id, responsavel_agencia_id),
  lancamento_expires_at (expiração automática de 30 dias — verificada
  sob demanda, NÃO é um cron job), snapshots em JSON de checklist/etapas/
  briefing. Tem ProjectTaskStage[], TaskBriefingAnswer[],
  TaskAttachment[], TaskAssignmentHistory[], ProjectCredential[].
- ProjectTaskStage: sub-etapas ordenadas de um ProjectTask, cada uma com
  seu próprio status/checklist e flag de dependência sequencial.
- TaskBriefingAnswer: respostas do cliente/empresa às perguntas de
  briefing de uma tarefa (arquivos/links em JSON).
- TaskAttachment: anexo de arquivo/link/referência/entrega numa tarefa.
- TaskAssignmentHistory: trilha de auditoria das tentativas de
  atribuição automática/manual de nômade (critérios, pontuação,
  resultado).
- SystemAlert: alerta operacional (ex.: "nenhum nômade encontrado",
  "tarefa atrasada"), com severidade e estado de leitura, gerado pelos
  serviços de auto-atribuição.

TAREFAS / MODELOS DE TAREFA (camada legada/paralela mais antiga)
- TaskTemplate: template ligado a uma Specialty, precificado com
  complexidade, margem de lucro, multiplicador de urgência. Tem
  TaskExecution[].
- TaskExecution: instância de tarefa executada ligando Project +
  TaskTemplate + Nomade, com pipeline de status, prioridade, avaliação/
  feedback. (Parece ser um modelo mais antigo/simples de execução de
  tarefa, distinto do pipeline ProjectTask mais novo.)

FINANCEIRO (Faturas / Carteiras / Saques / Despesas / Pagamentos)
- Invoice: documento de cobrança ligado a Company/Project (status
  pendente/pago/atrasado/cancelado).
- Payment: registro de pagamento sandbox/fake para um Project
  (payment_method: CARTAO_TESTE/PIX_FAKE/BOLETO_FAKE/CREDITOS/ALLKOINS;
  gateway: FAKE_SANDBOX/STRIPE/PAGARME/ASAAS — só FAKE_SANDBOX está de
  fato implementado, os outros são valores de enum reservados para
  integração futura).
- WithdrawalRequest: pedido de saque do nômade via PIX
  (aguardando_analise→pagamento_agendado→pagamento_efetuado/cancelado/
  reprovado).
- Expense: despesa operacional da própria Allka (categoria, tipo
  fixa/variável, recorrência, recurrence_id agrupando série recorrente,
  flag is_recurring_base — a geração da série parece ser feita em código
  de aplicação, não em job agendado).
- Wallet: conta de carteira unificada chaveada por (owner_type,
  owner_id) onde owner_type ∈ company/agency/nomad/partner/platform. Tem
  WalletLedger[].
- WalletLedger: lançamento estilo partida-dobrada (tipo, direção, valor,
  saldo antes/depois, idempotency_key único, reference_type/
  reference_id polimórfico para payment/invoice/withdrawal/commission/
  expense/project). Escrito via lib/wallet-service.ts.

SQUAD (plano de crédito pós-pago)
- SquadConfig: 1:1 com Company; configuração de plano pós-pago (limite de
  crédito, mínimo mensal, dia de fechamento, prazo de pagamento, status
  ativo/pausado/cancelado/inadimplente). Tem SquadCycle[].
- SquadCycle: instância de ciclo de faturamento mensal para um
  SquadConfig (total consumido, ajuste de mínimo, total faturado, link
  soft para invoice_id, status aberto/fechado/faturado/pago/atrasado/
  cancelado).

COFRE DO PROJETO (credenciais)
- ProjectCredential: guarda credenciais de demo/serviço de um projeto
  (nome do serviço, url, usuário, password_demo — explicitamente só demo),
  categoria, metadados de rotação/expiração/compartilhamento.
- ProjectCredentialAccessLog: log de auditoria das ações no cofre
  (criado/visto/copiado/compartilhado/revogado/rotacionado/editado/
  arquivado/excluído).

TERMOS E COMPLIANCE
- Term: documento de termos versionado (nível de aceite empresa/usuário,
  tipos de conta-alvo em JSON). Tem TermAcceptance[].
- TermAcceptance: registro de um User aceitando um Term (único por
  termo+usuário), com endereço IP.

CHAT
- Conversation: thread direta/grupo/suporte; tem ChatParticipant[],
  ChatMessage[].
- ChatParticipant: participação de usuário numa conversa (papel,
  entrou_em).
- ChatMessage: conteúdo da mensagem + remetente + flag de lida.

ALLKADEMY (LMS / treinamento)
- Course, CourseModule, Lesson, CourseEnrollment (progresso/conclusão).

RELATÓRIOS / ANALYTICS / CONFIG DE PLATAFORMA
- ReportConfig: config de controle de acesso por chave de relatório
  (tipos de conta/papéis permitidos, usuários permitidos/bloqueados,
  escopo GLOBAL/OWN_*_SCOPE/CUSTOM_USERS, permissões de export/filtro).
- UsageEvent: evento de analytics opt-in (page_view/login/project_start/

BACKGROUND JOBS: NÃO HÁ scheduler/fila real (sem node-cron/bullmq/agenda).
O que existe:
- Manutenção no boot: limpeza de datas inválidas do MySQL
  (clean-zero-datetimes.ts), roda uma vez ao iniciar o servidor.
- Lógica "fire-and-forget" disparada por requisições de API (não
  agendada): seleção automática de nômade (selecionar-nomade.ts),
  atribuição automática de líder (atribuir-lider.ts), geração de tarefas
  a partir de produtos (generate-tasks.ts), geração de código de tarefa
  (task-code.ts), log de atividade de projeto, serviço de carteira
  (createLedgerEntry idempotente/transacional).
- Expiração de 30 dias de ProjectTask.lancamento_expires_at é checada
  SOB DEMANDA (comparação inline ao ler/editar a tarefa) — não existe
  varredura periódica que expire tarefas automaticamente sozinha.
- Scripts avulsos em src/scripts/ (auditoria, backfill, seed, reparo) são
  utilitários de linha de comando rodados manualmente, não jobs
  agendados.

INTEGRAÇÕES EXTERNAS: quase nenhuma é real hoje —
- Pagamento: só FAKE_SANDBOX implementado; Stripe/Pagar.me/Asaas são só
  valores de enum reservados, sem SDK/cliente de API algum no código.
- CRM (Bitrix): só um campo bitrix_sync reservado, sem client/webhook.
- Armazenamento de arquivo: sem SDK de S3/Cloudinary/GCS — campos de
  imagem/anexo são LongText (provavelmente base64 ou URL externa gerida
  fora deste backend).
- E-mail/SMS/WhatsApp: nenhuma integração real (sem nodemailer/SendGrid/
  Twilio/Zenvia/WhatsApp Business API) em lugar nenhum do backend.
- Variáveis de ambiente realmente consumidas: DATABASE_URL, JWT_SECRET,
  PORT, NODE_ENV, FRONTEND_URL (CORS), SEED_TEST_USER_PASSWORD,
  ALLOW_TEST_USERS_SEED_IN_PRODUCTION — nenhuma chave de API de serviço
  externo está declarada no .env de exemplo.

CONCLUSÃO: o backend é hoje um sistema autocontido/sandbox (gateway de
pagamento fake, sem CRM/armazenamento/comunicação real ligados); o
schema/enums já preveem plugar Stripe/Pagar.me/Asaas e Bitrix no futuro,
mas essa ligação ainda não existe.

12. AUTENTICAÇÃO E CONTROLE DE ACESSO
================================================================================
- JWT stateless (sem sessão/cookie). Middleware verifyToken lê
  `Authorization: Bearer <token>`, verifica com JWT_SECRET, anexa
  `{ id, email, role, account_type }` em req.user.
- requireRole(...roles) é o middleware genérico de checagem de papel
  (usado majoritariamente para "admin").
- Login (POST /api/auth/login) checa e-mail/senha (hash bcryptjs),
  is_active, e um accessType opcional (ADMIN/AGENCY/NOMAD/COMPANY/
  PARTNER/LEADER) contra o role/account_type real do usuário — essa é a
  trava do seletor de painel na tela de login. Emite um JWT válido por
  7 DIAS.
- Modelo de papel: User.role é uma string livre (company_user/
  company_admin/company_financial/agency_user/agency_admin/nomad/admin/
  partner — mais "lider" usado em alguns middlewares locais).
  User.account_type é uma classificação paralela mais grosseira
  (empresas/agencias/nomades/admin/parceiro).
- Algumas rotas implementam middleware de papel PRÓPRIO/local em vez de
  reusar requireRole (ex.: requireLider em lider.ts, requireAdmin em
  habilidades.ts).
- Sem OAuth/SSO, sem refresh token (o JWT já é de longa duração, 7 dias).
- /api/share e /api/reports/usage-event são intencionalmente públicos
  (sem verifyToken).

No FRONTEND (RequireAuth em App.tsx):
- Sem token no localStorage → redireciona pro /login do prefixo da URL
  atual.
- Com token, mas o role/account_type não bate com o prefixo da URL →
  redireciona pro dashboard do PRÓPRIO perfil.
- Admin acessa qualquer coisa.
- Evento global `allka:unauthorized` (401 de qualquer chamada de API)
  redireciona pro login certo.
- TermAcceptanceGate bloqueia a UI inteira se houver termos pendentes,
  independente do papel.
- AccountTypeContext resolve "qual portal estou vendo" com esta ordem de
  prioridade: localStorage["dev_active_profile"] → legado
  localStorage["simulatedUser"] → inferido pela URL atual — isso é
  claramente um mecanismo de DEV/PREVIEW de troca de perfil, não o fluxo
  real de produção.
- Único ponto público sem autenticação nenhuma: /dashboard/share/:token.

  template).
- Tokens de cor em OKLCH definidos em globals.css (:root e .dark), mais
  um .dark.high-contrast para acessibilidade.
- Uma segunda camada de tema (marca/portal, separada de claro/escuro) é
  escrita como CSS vars pelo SidebarContext: --app-brand-gradient,
  --app-brand-header, --app-brand-button, --app-brand-active,
  --app-font-sidebar/heading/body/table, --sidebar-width etc.
- Cor de destaque por portal (convenção, não CSS var): Admin=rosa,
  Nômade=âmbar, Empresa=violeta, Agência=índigo, Parceiro=azul.
- Escala de fonte GLOBAL separada (useFontScale) — 7 níveis, seta
  document.documentElement.style.fontSize diretamente.
- Existem overrides de dark mode "hand-written" em globals.css que
  remapeiam classes utilitárias claras (bg-white, bg-slate-50 etc.)
  especificamente dentro de [data-slot="sheet-content"] (o próprio
  SlidePanel) e [data-slot="dialog-content"]/main — porque muito
  conteúdo foi escrito com classes claras fixas em vez de tokens
  semânticos.

14. COMUNICAÇÃO ENTRE USUÁRIOS, CHAT E NOTIFICAÇÕES — O QUE É REAL X MOCK
================================================================================
Esta é provavelmente a área mais importante de entender antes de pedir
ajustes a outro agente, porque tem bastante coisa "bonita na tela, mas
sem nada real por trás". Resumo por item:

| Área | Backend real? | Frontend ligado de verdade? | Observação |
|---|---|---|---|
| Chat (Conversation/ChatMessage) | Sim (/api/chat/*) | Parcialmente — o widget existe mas o formato de dados não bate com a API (campos `contact`/`unread_count` que a API nunca retorna) | Hoje, na prática, não funciona para conversas reais |
| SystemAlert (alertas operacionais) | Sim (/api/system-alerts/*) | Sim, sino no header, verificado a cada 60s | Alerta GLOBAL, não por usuário; só é criado quando o matching automático de líder/nômade falha |
| Alertas de agência (/api/agencies/me/alerts) | Sim, calculado ao vivo | Sim | Real, baseado em status de tarefa/projeto |
| "Central de Notificações" do admin (templates/regras/histórico) | NÃO existe | Mock (arrays fixos, setTimeout falso) | Maquete pura |
| Painel de preferências/caixa de notificação por usuário | NÃO existe | Mock | Maquete pura |
| Modais de notificação/e-mail/WhatsApp rápidos | NÃO existe | Mock (toast local) ou só abre wa.me | "Enviar pela Plataforma" nunca chama API de verdade |
| Briefing de tarefa (perguntas e respostas) | Sim (/api/project-tasks/:id/briefing, /submit-briefing) | Sim | Funcional de verdade; o "assistente de IA" que preenche o briefing colado é uma heurística LOCAL (separar frases/palavras-chave), NÃO é uma chamada de LLM real |
| Anexos de tarefa | Sim | Sim | Funcional |
| Comentários/thread de discussão em tarefa ou projeto | NÃO existe | NÃO existe | Não há model de Comment no banco; qualquer "conversa" além do briefing estruturado teria que ser via campo observations, anexos, ou o chat (que está quebrado) |
| Infra em tempo real (WebSocket/socket.io/pusher) | NÃO existe | NÃO existe | Tudo é polling (só o sino de alerta, a cada 60s) ou fetch pontual |
| E-mail transacional | NÃO existe | NÃO existe | Nenhum provedor, nenhum template, nada é realmente enviado |
| Workflow de aprovação (agência/cliente) | Sim (TaskExecution.status) | Só aparece via sino de alerta | Nenhum e-mail/push acompanha a mudança de status |
| Log de atividade/auditoria da empresa (aba "Logs") | NÃO existe | Mock (buildMockLogs gera dados fake determinísticos) | Não há model de audit log real |

CONCLUSÃO PRÁTICA: se for pedir a outro agente para "melhorar as
notificações" ou "consertar o chat", é importante deixar claro que hoje
NÃO existe infraestrutura real de e-mail/tempo-real por trás — qualquer
plano de ajuste provavelmente vai precisar CRIAR essa infraestrutura do
zero (escolher um provedor de e-mail, decidir entre polling mais
agressivo ou WebSocket, criar um model de Notification por usuário no
Prisma, e corrigir o mismatch de schema entre o ChatContext do frontend
e a API real de chat) — não é um ajuste cosmético pequeno.

16. JORNADAS DE PONTA A PONTA (como os perfis se conectam na prática)
================================================================================
Um fluxo típico, juntando as peças documentadas acima:

1. Um ADMIN cadastra o catálogo em admin/produtos (produtos, variações,
   add-ons) e vincula "modelos de tarefa" (admin/modelos-tarefas) a cada
   produto. Preços/comissões/impostos são configurados em
   admin/precificacao; especialidades em admin/especialidades.

2. Uma EMPRESA (ou uma AGÊNCIA em nome de uma empresa) monta um projeto:
   navega o catálogo (company/produtos ou agency/catalogo — mesmo
   componente ProductCatalogView), adiciona produtos ao carrinho
   (project-basket-context), e finaliza a contratação
   (ProjectCreateNewPanel), gerando um Project + vários ProjectProduct.

3. Ao contratar, o backend gera ProjectTask automaticamente a partir dos
   CatalogTask vinculados a cada produto (lib/generate-tasks.ts),
   populando checklist/etapas/perguntas de briefing como snapshot.

4. A AGÊNCIA (ou a própria empresa) preenche o briefing da tarefa
   (task-launch-drawer.tsx → /api/project-tasks/:id/briefing) e
   "lança"/libera a tarefa (submit-briefing) — isso dispara
   atribuir-lider.ts, que tenta achar automaticamente um LÍDER da área
   certa para revisar (ou cria um SystemAlert se não achar ninguém).

5. Em paralelo/depois, selecionar-nomade.ts tenta encontrar
   automaticamente um NÔMADE qualificado (por NomadeHabilidade/
   Qualification) para executar a tarefa — se não achar ninguém, o
   status vira AGUARDANDO_NOMADE e outro SystemAlert é criado.

6. O NÔMADE vê a tarefa em nomades/tarefasdisponiveis, se candidata, e
   ao ser designado passa a vê-la em nomades/minhastarefas. Executa,
   sobe anexos/entregas (TaskAttachment tipo "delivery"), e envia para
   revisão.

7. O LÍDER da área vê a tarefa em leader/qualificacao (ou
   leader/tarefas com os filtros de "briefings para revisar"/"entregas
   aguardando análise"/"atrasadas"), e aprova ou devolve com motivo
   (leader/devolvidas guarda o que ele já devolveu).

8. Após aprovação, o ciclo financeiro entra em ação: geração de Invoice
   (fatura) para a empresa, lançamentos em Wallet/WalletLedger para
   agência/nômade/parceiro conforme a comissão configurada, tudo
   visível em admin/financeiro (e nas versões "meu financeiro" de
   agência/nômade/parceiro).

9. Se a empresa/agência veio de uma indicação, o PARCEIRO responsável
   ganha comissão recorrente sobre o MRR gerado (PartnerCommission),
   visível em partner/comissoes, com saque disponível em partner/saques.

10. Ao longo de todo o processo, o único mecanismo de "avisar alguém que
    algo aconteceu" que de fato funciona hoje é o sino de alertas no
    header (polling a cada 60s, alimentado por SystemAlert + o cálculo
    ao vivo de alertas de agência) — não há e-mail, WhatsApp ou chat
    real disparado automaticamente por essas transições de status (ver
    seção 14 para o detalhe completo do que é mock).
